# 도메인 문서

엔지니어링 스킬들이 코드베이스를 탐색할 때 이 프로젝트의 도메인 문서를 어떻게 소비해야 하는지 설명합니다.

## 탐색 전에 읽을 것

- **`CONTEXT.md`** (repo root에 위치), 또는
- **`CONTEXT-MAP.md`** (repo root에 존재할 경우) — 컨텍스트별 `CONTEXT.md`를 가리킵니다. 관련 있는 각각의 파일을 읽으세요.
- **`docs/adr/`** — 작업 영역과 관련된 ADR를 읽으세요. 멀티 컨텍스트 리포의 경우 `src/<컨텍스트>/docs/adr/`도 확인하세요.

해당 파일이 없으면 **조용히 진행하세요.** 부재를 알리지 말고, 미리 생성을 제안하지 마세요. `/domain-modeling` 스킬(`grill-with-docs`와 `improve-codebase-architecture`를 통해 도달)은 용어나 결정이 실제로 확정될 때 lazily 생성합니다.

## 파일 구조

단일 컨텍스트 리포 (대부분의 리포):

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-이벤트-소싱-주문.md
│   └── 0002-쓰기-모델용-포스트gres.md
└── src/
```

## 용어집의 용어를 사용하세요

출력에서 도메인 개념을 명명할 때(이슈 제목, 리팩터링 제안, 가설, 테스트 이름 등), `CONTEXT.md`에 정의된 용어를 사용하세요. 용어집이 명시적으로 기피하는 동의어로 변경하지 마세요.

필요한 개념이 용어집에 아직 없다면, 이는 신호입니다 — 프로젝트에서 사용하지 않는 언어를 만들고 있거나(재고려 필요) 실제 공백이 존재하는 것입니다(`/domain-modeling`에 기록).

기존 ADR와 모순되는 출력이 있을 경우, 조용히 재정의하지 말고 명시적으로 표시하세요:
